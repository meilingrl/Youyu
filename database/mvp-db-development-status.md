# CampusMarket MVP 数据库开发状态说明

## 1. 当前阶段结论

当前数据库工作已经达到：

- **可支持前后端 `MVP` 并行开发**
- **可支持核心主链路接口建模与联调**
- **尚未达到完全冻结、不可再改的最终版**

换句话说，当前数据库已经具备 `MVP` 所需的核心实体、主外键关系、主要状态字段和初始化脚本，前后端可以基于本版表结构继续推进开发。

## 3. 对 MVP 开发的支持程度

### 3.1 已可直接支撑的开发

以下前后端 `MVP` 功能已经可以直接基于当前数据库推进：

- 用户注册、登录、基础资料维护
- 学生认证申请、审核结果展示
- 用户权限档位控制
- 地址管理
- 商品发布、编辑、上下架
- 商品分类、图片、资料资源管理
- 店铺申请、店铺审核、店铺信息展示
- 购物车
- 下单、订单列表、订单详情
- 统一订单主状态管理
- 模拟支付记录
- 退款记录
- 收藏与评价
- 举报、信用事件、风控限制
- 电子资料的预览与完整交付骨架

### 3.2 当前已能支撑到的联调层级

当前数据库已足够支撑：

- 后端 `DAO / JDBC / Service` 层开发
- 前端基于接口字段先做页面与状态展示
- 管理端列表、详情、审核、状态操作类接口
- 订单、支付、退款、审核等关键对象的基础流转开发

## 4. 当前仍未完全收口的部分

以下内容不会阻塞 `MVP` 开发启动，但仍需后续进一步细化：

- 各状态字段的完整流转规则尚未单独文档化
- “当前有效认证唯一”的精确实现方式还需在应用层明确
- 管理员权限粒度尚未细化到更完整的后台 RBAC 方案
- 信用等级的聚合口径尚未最终定义
- 逻辑删除后的归档、冷库、TTL 策略尚未细化到执行级别
- 初始化种子数据尚未生成

## 5. 当前设计边界

为避免超出 `MVP` 范围，当前数据库**有意不展开**以下内容：

- 真实商用支付体系
- 分账体系
- 复杂营销体系
- 优惠券正式体系
- 多商户结算体系
- 复杂售后单体系
- 复杂即时通讯体系

这意味着：

- 当前数据库是 `MVP` 可用版
- 不是最终商业化版本
- 但主链路结构已经具备后续扩展基础

## 6. 当前推荐协作方式

建议前后端开发以当前数据库版本作为：

- `MVP` 第一版联调基线
- 接口字段命名参考源
- 核心业务状态字段参考源

建议协作时遵守：

- 不随意改动主键/外键和核心状态字段命名
- 如果新增字段，优先加可空扩展字段，不轻易推翻现有结构
- 如果发现状态流转不清晰，先补规则说明，再改表结构

## 7. 当前阶段判断

当前数据库工作处于：

- **阶段：MVP 第一版可落地、可联调、可继续扩展**

不是：

- 最终冻结版
- 商业化完备版
- 无需再补规则说明的终稿
