# 用户与角色模型

## 设计原则

角色和状态必须分离。

- 角色回答“这个主体是什么类型的人”
- 状态回答“这个主体目前处于什么业务阶段”

如果把认证状态、卖家行为、店铺经营都当成并列角色，系统会被不必要地复杂化。

## 基础角色

### 游客

未登录用户。

可选能力：

- 浏览公开信息
- 查看部分商品
- 查看平台介绍

限制：

- 不能交易
- 不能评价
- 不能发布商品

### 学生用户

平台的基础业务角色。

学生用户在系统中既可能是买家，也可能是卖家。买卖双方不需要拆成两个独立前台身份。

### 平台管理员

负责后台治理、审核、用户管理、商品管理、订单管理、内容管理和数据看板。

### 合作商家

可选角色。仅在后期确认品牌合作方案后启用。

## 学生用户的业务状态

### 注册状态

- 未注册
- 已注册

### 认证状态

- 未认证
- 认证中
- 已认证
- 认证失败
- 认证失效

### 交易权限状态

- 无交易权限
- 可购买
- 可发布
- 可评价
- 受限中

### 经营状态

- 普通学生用户
- 带卖家标签的学生用户
- 已开通店铺的学生用户

### 信用状态

- 正常
- 观察中
- 受限
- 冻结

## 卖家和店主的区别

### 卖家

卖家不是独立角色。

定义：

- 认证学生用户发布过商品后，即具备卖家标签

特点：

- 不需要新建独立前台
- 不要求店铺主页
- 适合偶发性或轻量级售卖

### 店主

店主也不是基础角色，而是学生用户的一种经营升级状态。

定义：

- 学生用户满足条件后开通店铺，成为店主

特点：

- 拥有店铺主页
- 可沉淀店铺评分与简介
- 可配置活动公告、常客优惠等能力
- 更适合长期稳定经营

### 结论

卖家和店主的区别不在于“卖一件还是卖十件”，而在于是否具备持续经营意图、店铺化展示需求和更高信用沉淀能力。

## 学生认证原则

- 只有认证学生才能拥有完整交易权限
- 一个学生身份只能对应一个有效主账号
- 必须尽量防止重复认证和冒用

## 分层权益原则

平台采用“信息与信用分层换取权限”的原则。

核心思想：

- 用户提供越完整、越可信的信息，可获得越高的平台权限
- 用户积累越稳定、越正向的交易信用，可获得越高的经营能力
- 平台不强迫用户一次性交出全部信息，但必须让高风险能力建立在更高可信度之上

建议的分层结构如下：

### 第一层：游客

能力：

- 浏览平台介绍
- 浏览部分公开商品信息

限制：

- 不能购买
- 不能发布
- 不能评价

### 第二层：基础注册用户

能力：

- 完成基础账号注册
- 完善基础资料
- 收藏商品
- 进入个人中心

限制：

- 默认不开放高风险交易能力

### 第三层：基础买家

能力：

- 在完成基础信息填写后获得购买能力
- 可下单、收货、评价

说明：

- 基础买家不必立即完成最强学生认证
- 但必须达到平台定义的最低可信门槛

### 第四层：认证学生卖家

能力：

- 发布商品
- 编辑商品
- 管理个人发布内容

前提：

- 必须完成学生身份认证
- 必须通过唯一性校验

### 第五层：店铺经营者

能力：

- 开通店铺
- 获得更完整展示能力
- 扩大经营规模

前提：

- 学生身份认证通过
- 交易与信用达到门槛
- 无严重违规记录

开通原则建议采用：

- `基础门槛 + 主动申请 + 平台审核`

说明：

- 不是纯申请制，否则门槛过低
- 也不是纯自动升级，否则容易失控
- 用户需先满足基础信用与交易条件，再主动申请开店，由平台完成最终审核

建议的 `MVP` 开店门槛如下：

- 已完成学生认证
- 账号当前无风控限制
- 完成至少 `3` 笔有效交易
- 近阶段无严重举报成立记录
- 基础资料完善度达到平台要求

说明：

- `3` 笔有效交易是当前阶段的建议起点，便于控制门槛和实现复杂度
- 后续可按平台数据调整为更高门槛

该分层结构体现了“把信息权交给用户，同时给予相应回报”的产品原则。

## 认证方案候选

### 方案 A

- 学号
- 姓名
- 校园邮箱验证码

### 方案 B

- 学号
- 姓名
- 学院/专业信息
- 人工审核

### 方案 C

- 学号
- 学生证图片
- 人工审核

### 方案 D

- 年龄信息校验
- 基础实名信息

说明：

- 年龄可作为辅助约束
- 但年龄只能证明“可能是学生群体年龄段”，不能证明“当前真实在校”
- 因此年龄只能作为必要条件或风险参考，不能单独作为学生认证依据

### 方案 E

- 学信网或类似权威在学状态校验

说明：

- 从真实性角度看，这是高价值方向
- 但课程项目阶段需评估可接入性、开发复杂度和演示可行性
- 若无法真实接入，可在需求层预留为未来增强能力，而不作为 MVP 强依赖

当前建议：

采用“主认证 + 辅助校验 + 异常复核”的方案。

推荐组合如下：

- MVP 主认证优先采用 A 或 B
- 年龄信息作为辅助风险信号，不单独作为认证依据
- 学信网类权威校验预留为增强方向
- 必须体现“认证前后权限不同”和“身份不可重复认证”

## 推荐认证结论

### MVP 推荐方案

推荐采用：

- 基础注册信息
- 学号
- 姓名
- 学院/专业/年级
- 校园邮箱验证码或人工审核

系统同时应记录：

- 年龄或出生年月
- 注册设备或行为特征摘要
- 认证申请时间和认证结果

### 唯一性控制

必须支持：

- 一学号仅允许绑定一个有效主账号
- 认证通过记录不可重复使用
- 对异常重复申请进行拦截或人工复核

### 异常复核机制

平台必须承认“完全杜绝冒用很难”，但不能因此放弃防范。

因此应建立以下治理闭环：

- 举报入口
- 行为异常检测
- 二次身份核验
- 临时限制交易权限
- 信用降低
- 短期封号
- 严重情形永久封禁

### 风险判断信号

可作为异常识别参考的信号包括：

- 高频发布
- 短时间大量交易
- 被多人举报
- 资料信息矛盾
- 账号行为明显偏离普通学生使用特征

## 隐私保护原则

学生认证是高敏感能力，必须同步满足隐私保护要求。

要求包括：

- 仅采集必要信息
- 敏感信息最小展示
- 认证资料与公开资料隔离
- 后台仅授权管理员可见敏感认证字段
- 不在前台公开学号、证件图片等敏感数据
- 用户应能清楚知道“提供什么信息、换取什么权限”

## 校园信用体系

校园信用建议由以下指标共同构成：

- 认证状态
- 交易完成数
- 好评率
- 投诉与纠纷率
- 响应速度
- 交付稳定性
- 描述一致性
- 违规记录

信用体系用于：

- 帮助买家识别可靠卖家
- 激励长期认真经营
- 支撑店铺成长和平台治理

## 信用展示建议

当前建议采用：

- 底层保留信用事件记录
- 前台面向用户以等级制和状态制为主展示

说明：

- 不建议在前台过早暴露复杂显式分值
- 等级制更容易理解，也更适合课程项目与校园场景
- 后台和规则层仍可基于事件累积与内部权重计算信用结果

## 信用等级建议

`MVP` 阶段建议采用五级信用体系：

- `L0 新用户`
- `L1 已认证`
- `L2 稳定交易者`
- `L3 优质卖家`
- `L4 优质店主`

### 等级含义

#### L0 新用户

- 已注册但信用沉淀不足

#### L1 已认证

- 已完成学生认证
- 可视为具备基础可信身份

#### L2 稳定交易者

- 已完成一定数量的正常交易
- 无明显异常记录

#### L3 优质卖家

- 卖家表现稳定
- 好评与履约情况良好

#### L4 优质店主

- 店铺经营稳定
- 交易记录较完整
- 无严重违规

### 展示原则

- 前台以等级名称和少量说明展示
- 后台保留信用事件明细
- 平台规则层可继续使用内部权重计算，不要求前台暴露公式
