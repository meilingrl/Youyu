# 沟通与售后边界

## 1. 目的

本文档用于界定 `CampusMarket` 在当前阶段是否内置站内沟通，以及是否拆分独立售后单。

## 2. 站内沟通结论

当前阶段不构建复杂即时通讯系统。

建议采用：

- 交易留言
- 订单备注
- 必要的履约协商字段

当前不纳入范围：

- 完整 IM 聊天系统
- 已读未读会话体系
- 多媒体聊天

说明：

- 即时通讯成本高、耦合重、边界复杂
- 对课程项目而言，不应成为当前主阻塞项
- 平台重点应放在身份、交易、支付、履约和信用闭环

## 3. 售后建模结论

当前阶段不单独拆出“售后单”作为独立主实体。

当前使用以下对象承接即可：

- 订单 `Order`
- 退款记录 `RefundRecord`
- 举报 `Report`
- 风控限制记录 `RiskRestriction`

说明：

- 当前售后复杂度主要集中在退款、举报和争议处理
- 单独拆售后单会提前引入更多状态与流程复杂度
- 若后续增加更复杂的售后场景，再独立建模

## 4. 当前阶段适用边界

当前阶段的售后和沟通原则如下：

- 买卖双方以订单备注、交易留言和线下协商为主
- 平台保留退款、举报和风控介入能力
- 不在前期实现大而全的客服或 IM 系统

## 5. 非功能约束补充

虽然当前不做复杂沟通系统，但相关页面和流程仍必须满足以下要求：

- 留言、备注、退款申请、举报和售后处理入口必须有明确反馈，不能出现点击后长期无响应或状态不明。
- 会话失效、权限不足、商品下架、订单状态已变化等场景必须有统一提示，避免用户继续在错误状态下操作。
- 重复提交留言、重复发起退款、重复举报和非法状态流转应具备基础保护。
- 订单详情、退款处理和举报处理页在手机端与电脑端都应保持核心信息可读、关键操作可点。
- 即便不做完整聊天系统，也应保证沟通相关信息结构清晰，避免留言、备注和处理记录混杂在一起影响理解。

## 6. 设计影响

该边界决定后续设计应遵循：

- 订单详情页承担主要沟通与售后承载角色
- 退款、举报和限制状态应与订单状态、支付状态形成清晰分层
- 后台需具备最小治理入口，而不是把所有争议处理留给线下
- 当前阶段先保证治理闭环稳定成立，再考虑增强体验型沟通能力
