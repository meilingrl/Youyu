# 营销 MVP 范围

## 1. 目的

本文定义营销功能第一阶段的产品边界，供头 Agent 拆分子任务和验收使用。

本阶段目标是建立“店主发起、管理员审核、用户可见并可参与”的营销基础能力。实现细节由执行 Agent 在符合现有架构和测试要求的前提下决定。

## 2. 本阶段范围

### 2.1 优惠券

第一阶段优惠券必须形成基础闭环：

- 店主创建和管理自己店铺的优惠券。
- 优惠券发行前必须经过管理员审核。
- 用户可以主动领取已通过审核并处于可领取状态的优惠券。
- 用户可以查看自己已领取的优惠券。
- 订单预览和下单流程可以使用一张符合条件的优惠券。
- 下单时必须重新校验优惠券有效性，并记录实际优惠金额。
- 必须为领取超发、重复领取、重复使用等并发风险建立第一版防护。

第一阶段优惠券类型：

- 立减券。
- 满减券。

暂不支持折扣券、包邮券、叠加券、跨店券和自动最优券推荐。

### 2.2 店铺活动

第一阶段店铺活动做基础发布与展示闭环：

- 店主创建和管理自己店铺的活动。
- 活动发布前必须经过管理员审核。
- 通过审核且有效的活动可以在合适的前台入口展示。
- 活动可以关联店铺、时间范围、活动说明和展示状态。
- 活动可与优惠券并存，但第一阶段不要求活动直接驱动订单改价。

第一阶段店铺活动不做：

- 活动统计。
- 复杂活动规则引擎。
- 自动折扣计算。
- 满减、折扣、包邮等独立结算规则。

如执行 Agent 认为活动需要与优惠券联动，必须先在任务反馈中说明影响面，由头 Agent 决定是否纳入本轮。

## 3. 角色与权限

### 3.1 店主

店主负责发起店铺营销内容。

必须满足：

- 只能管理自己店铺的优惠券和活动。
- 不能审核自己提交的营销内容。
- 不能影响其他店铺的营销内容。

### 3.2 管理员

管理员负责营销内容审核与治理。

必须满足：

- 可以查看待审核优惠券和活动。
- 可以审核通过或驳回。
- 驳回时应保留可展示给店主的原因。
- 可以停用存在风险的营销内容。

### 3.3 用户

用户参与已审核通过的营销内容。

必须满足：

- 可以查看可领取优惠券。
- 可以领取优惠券。
- 可以查看自己的优惠券。
- 可以在结算时使用符合条件的优惠券。
- 可以在前台看到有效店铺活动。

## 4. 关键业务约束

- 营销内容以店铺为边界。
- 优惠券只能用于其所属店铺的订单。
- 当前订单主链仍遵守现有单卖家/单店铺边界。
- 下单必须以服务端重新计算结果为准。
- 历史订单的优惠金额不能被后续优惠券或活动修改影响。
- 活动展示不得暗示尚未实现的自动改价或统计能力。

## 5. 并发与一致性要求

第一阶段必须覆盖以下风险：

- 优惠券领取数量不能超过可发行数量。
- 同一用户不能绕过限制重复领取同一优惠券。
- 同一张用户券不能被重复使用。
- 订单创建失败时不能留下错误的优惠券使用状态。

本文不规定具体锁或数据库实现方式。执行 Agent 应选择与当前 MySQL/H2/JDBC 测试体系兼容的方案，并在反馈中说明一致性策略。

## 6. 前台入口要求

执行 Agent 应根据现有信息架构选择清晰入口，不要求新增多个重复入口。

必须覆盖：

- 买家查看和管理自己的优惠券。
- 买家在结算链路中选择优惠券。
- 用户可以在店铺或商品相关页面看到有效活动。
- 店主可以进入营销管理能力。

入口可以放在交易页、个人中心、店铺管理页或其他现有合适位置。执行 Agent 需在反馈中说明选择理由。

## 7. 后台入口要求

管理员必须有清晰的审核入口。

可以复用现有后台页面或新增营销审核页面。执行 Agent 需避免创建与现有后台导航冲突的重复入口，并在反馈中说明权限与导航处理方式。

## 8. 暂缓范围

以下内容不属于第一阶段：

- 活动效果统计。
- 平台级大促。
- 跨店优惠。
- 多券叠加。
- 自动最佳优惠选择。
- 复杂营销规则引擎。
- 支付网关改造。
- 定时任务体系重构。

## 9. 验收口径

本阶段验收以用户可完成的业务链路为准：

- 店主可提交优惠券和活动。
- 管理员可审核优惠券和活动。
- 用户可领取优惠券。
- 用户可查看自己的优惠券。
- 用户可在结算时使用一张有效优惠券。
- 有效店铺活动能在前台展示。
- 关键越权、重复领取、重复使用和超发场景被拒绝。
- 文档、HTTP 示例、API 说明和测试与实现保持一致。
