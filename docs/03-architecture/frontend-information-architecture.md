# Frontend Information Architecture

## 1. 文档目的

本文定义 `Youyu` 下一轮 UI/UX 重构的页面结构、路由分布、功能归属和导航原则。

它解决的问题是：哪些页面是主页面，哪些功能应该被收纳到哪个主页面里，哪些路由是用户旅程的一部分，哪些路由只是流程节点。

本文不定义具体视觉样式。视觉、动效、响应式与 AI 执行约束见 `ui-ux-constitution.md`。

## 2. 总体结构原则

### 2.1 前台不是功能列表

前台一级结构不再按“我的收藏 / 我的发布 / 购物车 / 订单”这种功能清单展开，而是按用户旅程组织：

1. 认识平台。
2. 探索商品和店铺。
3. 查看详情并交易。
4. 沟通。
5. 管理自己的身份、交易和偏好。

### 2.2 首页轻量化

首页减少信息量，不承担完整商品列表职责。

首页只负责：

- 建立第一印象。
- 说明平台是什么。
- 提供进入探索页的入口。
- 少量展示精选内容。
- 展示平台规模、信任与校园属性。

首页不应出现当前这种强顶部导航栏，也不应变成商品流页面。

### 2.3 探索页承载市场浏览

商品浏览、搜索、筛选、热搜、热门、精选店铺等主要浏览功能集中到探索页。

探索页是用户进入商品详情和店铺详情的主入口。

### 2.4 交易与消息都是一级能力

交易作为一级入口，统一承载购物车、订单、售后、支付状态、评价。

消息作为一级入口，统一承载用户对用户、用户对店铺、用户对客服、群聊或粉丝群等沟通场景。

### 2.5 店主不是管理员

后台只表示系统管理员能力。

店主是前台用户身份的一种增强状态。店主不进入 `/admin`。店主的管理能力属于前台个人域或店铺域。

### 2.6 店铺是店主的对外主体

普通用户以个人主页作为主体。

当用户成为店主后，店铺成为其主要对外主页，个人信息被包含在店铺主体内。店主仍然管理自己的信息，但管理内容会扩展为店铺资料、商品、订单和店铺数据。

## 3. 前台一级页面

### 3.1 首页

建议路由：`/app/home`

页面职责：

- 平台介绍。
- 轻量主视觉。
- 进入探索页的搜索或引导入口。
- 少量精选商品或精选主题卡片。
- 少量精选店铺展示。
- 平台范围、用户数、店铺数、交易或商品规模等宣传数据。
- 学生认证、交易保障、举报治理等信任说明。
- 买家与卖家入门引导。

首页不承担：

- 完整商品列表。
- 复杂筛选。
- 多层功能导航。
- 订单、收藏、发布等个人功能入口堆叠。

### 3.2 探索页

建议路由：`/app/explore`

可接受命名：`探索`、`发现`、`市场`。当前讨论中，“市场”可用但略俗，优先考虑“探索”或“发现”。

页面职责：

- 商品搜索。
- 分类和场景入口。
- 高质量筛选。
- 商品网格。
- 热搜、热门关键词、最近上新。
- 精选店铺穿插展示。
- 进入商品详情与店铺详情。

探索页是 Etsy 式整体市场体验的核心页面，搜索筛选框应学习 Airbnb 的分段式、聚焦式、可扩展体验。

### 3.3 商品详情页

建议路由：`/app/products/:id`

页面职责：

- 商品图与媒体展示。
- 商品标题、价格、类别、成色或状态。
- 加购、购买、收藏、联系卖家。
- 交易方式与平台保障。
- 卖家或店铺入口。
- 商品描述与细节。
- 评价与相关推荐。

商品详情页应借鉴 Etsy 的交易决策结构和 Aesop 的静态图文质感。

### 3.4 店铺详情页

建议路由：`/app/shops/:id`

页面职责：

- 店铺对外主页。
- 店主身份信息的精选展示。
- 店铺介绍。
- 店铺商品。
- 店铺评价。
- 店铺信用与交易数据。
- 联系店主。
- 关注、粉丝群、优惠群入口。

当用户是店主时，店铺页同时承担其对外身份主体；个人信息不应与店铺信息割裂成两个互相竞争的主页。

### 3.5 交易中心

建议路由：`/app/trade`

页面职责：

- 购物车入口。
- 订单列表。
- 订单详情。
- 支付状态。
- 售后、退款、举报入口。
- 待评价。
- 我的评价。

交易中心是用户交易闭环的主入口。购物车可以保留独立路由和高频入口，但在信息架构上属于交易域。

### 3.6 消息中心

建议路由：`/app/messages`

页面职责：

- 用户对用户会话。
- 用户对店铺或店主会话。
- 用户对客服会话。
- 群聊、粉丝群、优惠群。
- 交易关联会话。
- 售后关联会话。

消息中心是平台基础设施，不应只作为订单备注或商品详情里的局部能力。

### 3.7 个人主页

建议路由：`/app/me`

页面职责：

- 头像、昵称、认证状态。
- 校园身份。
- 个人统计。
- 交易摘要。
- 收藏摘要。
- 评价摘要。
- 店主状态。
- 若用户是店主，展示店铺管理入口、店铺数据、对外店铺主页入口。
- 若用户不是店主，展示开店、发布或认证引导。

个人主页不承载偏好设置、账号安全、地址管理等配置型功能。

### 3.8 设置中心

建议路由：`/app/settings`

页面职责：

- 个人偏好设置。
- 地址管理。
- 账号安全。
- 通知设置。
- 隐私设置。
- 退出登录。

个人主页回答“我是谁”，设置中心回答“我要改什么配置”。

## 4. 前台建议路由

### 4.1 主路由

```text
/app/home
/app/explore
/app/products/:id
/app/shops/:id
/app/trade
/app/messages
/app/messages/:conversationId
/app/me
/app/settings
```

### 4.2 交易域路由

```text
/app/cart
/app/checkout
/app/payments/:orderId
/app/orders
/app/orders/:id
/app/reviews/pending
/app/reviews/mine
```

### 4.3 设置域路由

```text
/app/settings/preferences
/app/settings/addresses
/app/settings/security
/app/settings/notifications
/app/settings/privacy
```

### 4.4 店主域路由

店主能力属于前台，不属于 `/admin`。

```text
/app/shop/manage
/app/shop/manage/products
/app/shop/manage/publish
/app/shop/manage/orders
/app/shop/manage/profile
```

### 4.5 兼容策略

现有路由可以先保留并逐步迁移：

| 现有路由 | 建议归属 |
| --- | --- |
| `/app/products` | 迁移或别名到 `/app/explore` |
| `/app/profile` | 迁移或别名到 `/app/me` |
| `/app/preferences` | 迁移到 `/app/settings/preferences` |
| `/app/seller/products` | 迁移到 `/app/shop/manage/products` |
| `/app/seller/publish` | 迁移到 `/app/shop/manage/publish` |
| `/app/orders` | 保留，属于交易域 |
| `/app/cart` | 保留，属于交易域 |

## 5. 前台导航原则

### 5.1 桌面端

首页不使用特别突出的传统顶部导航栏。

建议使用轻量浮动入口：

```text
Youyu / 搜索 / 发布 / 消息 / 个人
```

探索页、交易页、消息页、个人页可以使用更明确的局部导航，但仍避免把所有功能平铺在顶部。

### 5.2 移动端

移动端可以使用底部导航：

```text
首页 / 探索 / 消息 / 交易 / 我的
```

购物车可作为交易入口中的高频动作，也可以在探索页或商品详情页中出现独立图标入口。

### 5.3 导航命名

一级导航推荐：

```text
首页 / 探索 / 消息 / 交易 / 我的
```

其中“探索”也可讨论为“发现”。“市场”语义明确，但气质略普通，不作为首选。

## 6. 后台页面结构

后台专指系统管理员，不包括店主。

### 6.1 后台一级域

```text
总览
用户与认证
市场内容
交易治理
搜索与运营
客服与消息
系统设置
```

### 6.2 后台建议路由

```text
/admin/dashboard
/admin/users
/admin/verifications
/admin/products
/admin/shops
/admin/orders
/admin/reports
/admin/hot-search
/admin/support
```

现有后台基本结构可保留，但需要补充客服与消息治理入口。

### 6.3 客服与消息治理

建议先建立 `/admin/support`，后续按复杂度拆分：

```text
/admin/support/conversations
/admin/support/after-sales
/admin/support/groups
```

职责：

- 客服会话。
- 售后协助。
- 举报或纠纷关联会话。
- 群聊、粉丝群、优惠群治理。
- 敏感消息、异常消息、待处理会话。
- 会话状态管理。

## 7. 并行开发边界

为便于多个 Agent 并行工作，应按页面域拆分任务，并尽量避免多个任务同时修改同一组共享文件。

推荐拆分：

- Shell 与设计 token：只由一个 Agent 负责，先完成或作为共享基础任务。
- 首页与探索页：一个 Agent 负责前台入口和浏览主线。
- 商品详情与店铺详情：一个 Agent 负责交易决策和店铺主体。
- 交易与评价：一个 Agent 负责购物车、订单、售后、评价。
- 消息与客服入口：一个 Agent 负责消息中心和后台客服入口。
- 个人、设置与店主身份：一个 Agent 负责个人域和店主域。
- 后台导航与支持页：一个 Agent 负责管理员侧信息架构。

共享文件如 `frontend/src/styles/*`、`frontend/src/router/*`、`frontend/src/constants/navigation.js` 应由基础任务或集成任务统一修改，其他任务只在必要时提出需求，避免并行冲突。

## 8. 当前结论

当前页面结构决策：

- 首页轻量化。
- 一级入口采用首页、探索、消息、交易、我的。
- 探索页承载商品、搜索、筛选、精选店铺、热搜热门。
- 商品详情和店铺详情保留为核心详情页。
- 交易作为一级能力。
- 消息作为一级能力。
- 个人主页和设置中心分离。
- 店铺是店主的对外主体，店主能力属于前台。
- 后台只表示系统管理员，补充客服与消息治理。
