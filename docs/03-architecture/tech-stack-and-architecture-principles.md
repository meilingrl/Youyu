# 技术栈与架构原则

## 1. 课程约束优先

项目实现必须优先满足课程明确要求：

- 前端：`HTML`、`CSS`、`JavaScript`、`Vue 3`
- 前端增强：`Element Plus`、`Pinia`
- 后端：`Java`
- 课程技术点：`Servlet`、`Filter`、`Listener`、`JDBC`
- 框架：`Spring`、`MyBatis`
- 数据库：`MySQL`

## 2. 当前运行基线与目标路线

本节区分“当前已落地运行基线”和“课程/后续目标路线”。

当前阶段不得把 MySQL/MyBatis 目标误读为必须立即全量迁移。除非任务明确要求迁移，否则新增 MVP 收口工作应优先沿用当前 H2 file + JDBC 基线。

### 2.1 前端

- `Vue 3`
- `Vue Router`
- `Pinia`
- `Element Plus`
- 统一 `api` 封装
- 统一 `layout`、`views`、`components`、`stores` 结构

### 2.2 后端

- 基于 `Spring Boot` 组织应用骨架
- 通过清晰分层体现课程技术点职责
- 使用 `Filter` 处理请求上下文、鉴权或链路辅助
- 使用 `Listener` 处理启动日志、基础初始化或监听行为
- 当前运行基线使用 `JDBC` / `JdbcTemplate` / mapper 实现承载正式持久化能力
- `MyBatis` 是课程约束中的目标技术点，后续引入时必须有独立任务说明迁移范围、兼容策略和测试要求

### 2.3 数据库

- 当前默认运行数据库：H2 file，路径为 `~/.campusmarket/h2/campusmarket`
- 当前测试数据库：H2 in-memory
- 当前 H2 配置使用 MySQL compatibility mode，便于后续迁移审计
- 目标数据库方向：`MySQL`
- `database/001_mvp_init.sql` 是 MySQL 8 设计资产和迁移参考，不是当前默认启动脚本
- 后续可通过 MySQL profile、Flyway 和 MyBatis mapper 逐步迁移，但必须以独立任务或 ADR 明确决策

### 2.4 当前阶段选择原则

- 小体量 MVP 收口任务：沿用 H2 file + JDBC
- 新增持久化表：优先在现有 `schema.sql` 风格下做兼容性设计，并同步数据库文档
- MySQL/MyBatis/Flyway：仅在专门迁移任务中引入
- 后端集成测试：默认继续使用 H2，确保本地和 CI 可快速运行

## 3. 当前阶段架构原则

### 3.1 先稳住单体，再讨论拆分

当前目标是交付稳定单体应用，不引入微服务复杂度。

### 3.2 先稳主链，再扩新能力

优先级必须是：

1. 认证
2. 商品与审核
3. 订单与支付
4. 举报与治理
5. 偏好与统计增强

### 3.3 先真实数据，再继续堆页面

当前阶段必须把：

- 内存实现
- mock 数据
- 临时 Map 结构

逐步收敛为：

- 正式 Mapper/Repository
- 正式 DTO
- 正式数据库模型

### 3.4 允许双轨开发，但主轨优先

当前阶段允许并行推进：

- 主轨：接口收口、数据持久化、核心链路联调
- 副轨：偏好设置、统计页壳子、指标定义、低耦合增强

但副轨不得阻塞主轨。

## 4. 接口设计原则

- 统一响应结构
- 统一错误码语义
- 统一状态枚举命名
- 统一审计字段
- 核心请求逐步从 `Map<String, Object>` 收敛为 DTO

## 5. 鉴权原则

- 认证与鉴权必须区分
- 菜单隐藏不是权限控制的全部
- 后端接口必须做真实权限校验
- 学生认证状态、卖家权限、店铺能力应分层判断

## 6. 支付实现原则

- 当前 `MVP` 采用模拟支付
- 支付模块必须按可替换网关方式设计
- 后续可选接入支付宝沙箱
- 不以真实商用支付上线为当前目标

## 7. 数据持久化原则

当前阶段必须优先替换以下内存能力，且默认使用当前 JDBC mapper 分层完成：

- 用户与认证
- 商品与审核
- 店铺与店铺申请
- 购物车、订单、支付、退款
- 举报与信用记录

允许后置但要预留：

- 浏览历史
- 用户统计快照
- 店铺统计快照

当前阶段已经进入 MVP 收口，因此以下能力不应继续只停留在内存或纯预留状态：

- 偏好设置
- 用户统计真实聚合或明确不可用状态
- 店铺统计真实聚合或明确不可用状态
- 电子商品访问留痕

## 8. 统计与偏好能力接入原则

### 8.1 统计能力

- 优先基于真实业务表聚合
- 当前阶段允许先用查询拼装，不强制先建复杂分析库
- 指标必须可解释

### 8.2 偏好能力

- 与交易主链低耦合
- 优先抽为独立设置模型
- 前端主题能力应建立在统一设计变量体系上

## 9. 工程质量底线

- 统一接口风格
- 统一异常处理
- 统一日志与调试边界
- 不接受大规模复制粘贴式开发
- 不允许把“mock”“演示版”“测试账号”等内部标识直接展示给最终测试页面
