# MVP 范围冻结

## 1. 当前阶段目标

当前 `MVP` 的核心目标不是继续横向堆功能，而是交付一个：

- 完成度高
- 风格统一
- 核心链路真实成立
- 数据可持久化
- 结构易扩展
- 工程上可继续维护

的校园学生交易平台。

## 2. MVP 总体策略

当前 `MVP` 采用以下策略：

- 骨架已完成，当前进入收口阶段
- 真实链路优先于页面数量
- 持久化优先于继续扩展 mock
- 统一接口优先于局部快速拼接
- 低耦合新功能允许并行预研

## 3. 当前 MVP 必做闭环

### 3.1 用户与认证闭环

必须支持：

- 注册
- 登录
- 退出登录
- 学生认证申请
- 认证状态查看
- 后台认证审核
- 认证通过后权限提升

成功标准：

- 未认证用户可以浏览
- 基础用户可以购买
- 认证学生可以发布商品
- 达标认证用户可以申请店铺

### 3.2 商品发布与审核闭环

必须支持：

- 卖家发布商品
- 编辑商品
- 商品上下架
- 商品列表展示
- 商品详情展示
- 资料类商品进入待审核
- 管理员审核通过或驳回

成功标准：

- 资料类商品不能绕过审核直接公开上架
- 驳回必须有原因反馈

### 3.3 购物与交易闭环

必须支持：

- 购物车
- 订单预览
- 创建订单
- 模拟支付
- 订单状态流转
- 订单详情
- 确认收货
- 退款记录基础能力

当前支持的履约方式：

- 物流寄送
- 线下交付
- 电子交付

### 3.4 店铺闭环

必须支持：

- 店铺申请
- 店铺审核或开通
- 店铺主页基础展示
- 店铺商品列表

### 3.5 平台治理闭环

必须支持：

- 举报入口
- 后台举报处理
- 基础信用展示
- 基础风控限制记录

## 4. 当前 MVP 必做工程目标

### 4.1 数据与接口

必须完成：

- 核心数据持久化
- 统一接口响应结构
- 核心 DTO 收口
- 统一状态枚举
- 统一审计字段

### 4.2 质量与运行

必须完成：

- 前端可构建
- 后端可编译可启动
- 数据库初始化脚本可执行
- CI 能跑通基础校验

### 4.3 页面与体验

必须完成：

- 前台核心页面可用
- 后台核心页面可用
- 手机与电脑主链路可访问
- 空状态、错误状态、加载状态具备基本可用性

## 5. 当前 MVP 允许并行预研但不要求立即闭环的能力

### 5.1 用户统计

可以预研：

- 累计消费金额
- 已购商品数
- 最近浏览
- 收藏偏好
- 常购分类

约束：

- 第一阶段可以先做页面壳子、接口预留、指标定义
- 真实数据聚合以主轨持久化完成后接入

### 5.2 店铺统计

可以预研：

- 本月销售额
- 本月订单数
- 热销商品
- 浏览量
- 复购用户数

### 5.3 偏好设置

可以预研：

- 主题色
- 昼夜模式
- 默认地址
- 默认交付方式
- 默认排序与筛选
- 通知与展示偏好

约束：

- 偏好设置属于低耦合能力，可与主轨并行
- 但不得挤占核心链路持久化的主优先级

## 6. 当前阶段明确暂不纳入正式闭环的能力

当前不作为正式验收目标：

- 真实商用支付上线
- 复杂即时通讯
- 独立售后单体系
- 合作品牌正式入驻
- 复杂优惠营销体系
- 推荐算法
- 强 DRM 防复制
- 多商户分账
- 微服务拆分

## 7. 当前阶段成功标准

当前阶段完成的标志不是“功能很多”，而是：

- 核心链路真实落库
- 前端主要页面不再依赖主 mock 数据源
- 后台审核与治理不是空壳
- 状态流转一致
- 代码结构可继续承载后续偏好设置、统计能力和增强运营能力
