# 功能完成路线图

本文档只记录“产品功能还差什么、先做什么、为什么这样排”。上线基础设施、安全、合规、监控、部署和验收工作不写在这里，统一进入 `launch-preparation-roadmap.md`。

## 排期原则

- 先补已经被前端、HTTP 示例或 API 文档承诺的断点，再做新增能力。
- 同一阶段尽量只围绕一个业务闭环，避免多个任务同时改订单、用户、后台或统计口径。
- 数据可视化先补采集与指标口径，再补图表，避免页面和接口各算各的。
- 个性化依赖偏好、行为和通知基础，不抢在数据采集之前做复杂推荐。
- 支付升级会影响订单、退款和安全验收，应独立成阶段，不混入营销或客服任务。

## Phase F0: 契约断点与基础功能补齐

优先级：P0
建议周期：1-2 周
目标：先修复已经被前端、HTTP 示例或用户路径引用，但后端/页面未闭合的功能。

任务清单：

- 收藏功能闭环：新增收藏表、后端接口、权限校验、收藏计数一致性、前端收藏页联调。
- 分类接口与后台分类管理：补 `GET /api/categories`，再补后台分类增删改启停，商品发布继续复用同一分类源。
- 地址完整管理：在现有地址列表、创建、设默认基础上补编辑、删除和设置页入口。
- 支付展示口径修正：用户可见页面不再出现“模拟支付”等内部实现文案，保留开发文档说明。
- 评价分布真实化：商品/店铺评分分布改为真实聚合，移除 summary distribution stub。
- API/HTTP 对齐：清理仍指向不存在接口的 HTTP 示例或补齐对应实现。

完成条件：

- 前端现有入口不再调用缺失接口。
- `docs/06-http/` 和 `docs/09-api-spec/` 与实现一致。
- 前端测试、后端测试和关键手工链路覆盖收藏、分类、地址与评价 summary。

## Phase F1: 客服、售后与交易状态深化

优先级：P1
建议周期：2-3 周
目标：把客服后台正在推进的能力放进交易和售后闭环，避免独立客服页与订单/举报/调解割裂。

任务清单：

- 客服后台深化：会话/工单队列、处理状态、负责人、客服备注、与订单/举报/调解的上下文关联。
- 用户侧客服入口联动：商品、店铺、订单、售后入口进入同一客服上下文，不新建平行消息体系。
- 售后细化：退款驳回、退款协商记录、证据材料、处理原因、用户可见进度。
- 物流与线下交付细化：卖家侧发货入口、物流信息修改限制、线下交付双方确认历史。
- 通知收口：客服、售后、退款、发货、调解决策进入统一站内通知。

完成条件：

- 客服、售后、调解共享订单和举报上下文，不重复建业务真相。
- 后台客服页不只展示 placeholder，至少能处理一条真实队列。
- 交易状态变更均有用户可见反馈和后台留痕。

## Phase F2: 支付功能升级

优先级：P1
建议周期：2-3 周
目标：在不推翻现有订单状态机的前提下，把平台内 mock 支付升级为可替换、可失败、可回调的支付能力。

任务清单：

- 支付网关抽象真实化：支付方式、支付渠道、支付请求、回调验签接口分层。
- 支付状态补齐：失败、取消、超时、重复支付、重试、订单关闭。
- 支付宝沙箱接入：下单、跳转/二维码、异步回调、同步返回、验签。
- 退款链路升级：退款申请、退款审核、退款回调或后台确认、支付记录与退款记录一致性。
- 支付安全：幂等键、金额服务端校验、回调重放保护、敏感字段脱敏。

完成条件：

- mock 网关保留为本地/dev 可替换实现，真实网关逻辑不污染订单服务。
- 支付和退款的异常状态有后端测试和 HTTP smoke。
- 订单、支付记录、退款记录在重复提交和回调重试下保持一致。

## Phase F3: 数据采集、分析与可视化

优先级：P1
建议周期：3-4 周
目标：先建立可解释的数据采集和指标口径，再建设用户侧、店铺侧和后台侧可视化。

任务清单：

- 行为采集：商品浏览、店铺访问、收藏、搜索、购买、退款、客服/售后事件。
- 指标口径：GMV、订单量、转化漏斗、热门商品、分类销量、用户偏好、店铺经营指标。
- 后台运营大盘：销售趋势、订单趋势、商品排行、分类分布、用户活跃、治理信号。
- 用户侧统计：消费金额、购买件数、最近浏览、收藏偏好、常购分类。
- 店铺侧经营面板：本月销售额、本月订单数、热销商品、浏览量、收藏量、复购用户。
- 图表组件：折线图、柱状图、环图、排行表，并明确数据源和时间范围。

完成条件：

- 图表只展示真实聚合或明确标注为不可用，不做装饰性假图。
- 指标定义写入需求或 API spec。
- 后台、用户和店铺三类可视化复用同一指标口径。

## Phase F4: 个性化与偏好能力

优先级：P2
建议周期：2-3 周
目标：把偏好设置从“可保存”升级到“真实影响体验”，并为推荐个性化提供稳定输入。

任务清单：

- 主题能力真实化：系统/浅色/深色模式、主题色应用边界、未支持选项下线或降级。
- 通知偏好：订单、评价、客服、营销、系统通知按类型控制。
- 隐私设置入口：浏览/推荐/营销偏好开关，与合规同意记录区分清楚。
- 首页和搜索偏好：默认排序、默认履约方式、默认地址在对应流程真实生效。
- 推荐个性化增强：基于浏览、收藏、购买和类目偏好补充推荐理由与降级策略。
- 多语言评估：如果纳入范围，先做文案资源结构，再决定是否实现中英文切换。

完成条件：

- 设置页展示的每个选项都能真实影响至少一个用户路径，或明确移出 UI。
- 偏好 API、前端状态、localStorage 之间职责清楚。
- 个性化推荐在无数据、少数据、关闭偏好时都有可解释降级。

## Phase F5: 运营、营销与后台扩展

优先级：P2
建议周期：3-4 周
目标：在基础交易、支付和数据口径稳定后，再扩展运营能力，避免高并发营销逻辑反复改订单。

任务清单：

- 营销扩展：活动统计、复杂活动规则、折扣/满减/包邮、叠券、跨店券、自动最优券。
- 后台批量操作：批量下架商品、批量处理举报、批量审核或批量状态变更。
- 系统配置管理：热搜权重、推荐参数、营销开关、通知模板、客服队列配置。
- 店铺运营增强：店铺公告、标签、粉丝/关注、常客优惠、经营策略配置。
- 风控运营：异常账号二次核验、黑名单/限制策略、营销滥用监控。

完成条件：

- 营销和批量操作均有后端权限控制、审计日志和边界测试。
- 高并发发券、用券、活动改价不破坏订单金额快照。
- 系统配置有默认值、回滚路径和操作留痕。

## Phase F6: 全局 UI/UX 与移动端功能收口

优先级：P3
建议周期：2 周
目标：在主要功能闭合后统一体验细节，避免先打磨后返工。

任务清单：

- 设置中心信息架构收口。
- 非交易主流程的移动端适配。
- 空状态、错误状态、加载状态统一。
- 跨页面文案、导航层级、按钮命名一致性检查。
- 前端包体积按真实指标做最后一轮针对性治理。

完成条件：

- 桌面和移动关键路径完成截图或人工验收。
- 不再展示“待接入”“模拟”等会影响用户验收的内部词。
- UI 收口不引入新的业务状态或接口契约。

## 建议拆分方式

- F0 可拆为 4 个小任务：收藏、分类、地址、评价/契约对齐。
- F1 与正在做的客服后台保持同一任务组，不把客服、售后、调解拆成三个互相抢字段的任务。
- F2 支付升级独立成任务组，避免与营销、退款、合规同时改支付记录。
- F3 先做数据采集和指标 API，再做图表页面。
- F4 个性化只依赖 F3 的行为数据，不反向改 F3 指标口径。
- F5 高级运营等前面订单、支付、统计稳定后再启动。

## 与上线准备路线的边界

- 本文档负责“用户和管理员能做什么”。
- `launch-preparation-roadmap.md` 负责“系统如何安全、稳定、可部署、可监控地上线”。
- 如果一个任务同时改变功能和上线基础设施，应拆成两个任务，并在任务文档中互相引用。
