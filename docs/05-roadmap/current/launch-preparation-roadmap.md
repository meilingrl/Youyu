# 上线准备路线图

本文档只记录上线所需的基础设施、安全、合规、性能、部署、监控和验收工作。已完成的功能路线图已经归档；如需核对当前上线候选功能基线，以 `stage-roadmap.md` 和相关归档任务/路线记录为准。

## 上线准备边界

属于本文档：

- 安全加固、密钥、CORS、输入验证、依赖扫描。
- 隐私合规、用户协议、Cookie 同意、用户权利。
- 数据库、缓存、对象存储、备份、日志、健康检查。
- Docker、Nginx、CI/CD、生产配置、回滚方案。
- 性能压测、监控告警、上线验收和发布演练。

不属于本文档：

- 收藏、分类、地址、客服、支付升级、数据可视化、个性化、营销扩展等产品功能。
- 这些功能的历史顺序和拆分见已归档的功能路线与任务记录。

## 当前上线基线

截至 2026-05-30：

- 本地/dev/test 已有 Spring Boot + Vue + MySQL/H2 基线。
- `/api/health` 已存在基础健康检查。
- CI 已覆盖后端测试、前端测试、前端构建和 Playwright smoke。
- 非 dev/seed/test 环境下默认 JWT secret 已有 fail-fast 保护。
- 尚未完成生产级 HTTPS、CORS 白名单、部署容器、备份恢复、监控告警、合规同意和发布 runbook。

## Phase L0: 上线范围冻结与任务拆分

优先级：P0
建议周期：2-3 天
目标：先明确这条路线上只做“上线准备”，不把产品功能塞进基础设施任务。

代办清单：

- [ ] 确认上线候选功能范围，引用当前 `stage-roadmap.md` 和相关归档任务中已经验收的功能基线。
- [ ] 为安全、合规、部署、监控、备份分别创建 active task。
- [ ] 建立上线阻断清单：P0 blocker、可延期项、演示环境可接受项。
- [ ] 定义验收环境：本地、CI、预发布、生产或演示环境。
- [ ] 明确配置来源：环境变量、配置文件、密钥管理、示例配置。

完成条件：

- 每个上线准备方向都有唯一任务文档，不与功能任务混写。
- `CHANGELOG.md`、任务文档和 roadmap 的责任边界一致。

## Phase L1: 安全加固

优先级：P0
建议周期：1 周
目标：消除生产或公开演示环境中的基础安全风险。

代办清单：

- [ ] HTTPS 终止方案：Nginx/网关配置、证书来源、HTTP 到 HTTPS 跳转。
- [ ] JWT 密钥管理：生产环境必须从环境变量或密钥服务注入，禁止使用 dev default。
- [ ] CORS 白名单：按环境配置允许域名，禁止生产 `*`。
- [ ] 输入验证审查：注册、登录、商品、订单、支付、客服、营销、后台操作补齐 Bean Validation 或服务端校验。
- [ ] SQL 注入防护审查：确认动态 SQL 参数化，重点检查搜索、后台筛选、排序字段。
- [ ] 权限回归：用户、店主、客服、审核员、运营、订单管理员、超级管理员的后端权限矩阵。
- [ ] 敏感信息脱敏：日志、API 响应、错误信息不泄露密码、token、手机号、证件/学号全量值。
- [ ] 依赖与配置扫描：前后端依赖漏洞、提交密钥、生产配置弱默认。

完成条件：

- 安全 smoke 和权限回归有可复现命令。
- 生产 profile 无默认密钥、宽松 CORS 或显式 mock 支付暴露。
- 高风险管理端 mutation 均有后端权限控制和审计记录。

## Phase L2: 隐私合规与用户权利

优先级：P0
建议周期：1 周
目标：满足公开上线或演示时的用户告知、同意和数据权利基础要求。

代办清单：

- [ ] 隐私政策页面。
- [ ] 用户协议页面。
- [ ] Cookie 政策页面。
- [ ] Cookie 同意横幅和偏好管理。
- [ ] 注册流程加入协议同意校验。
- [ ] 用户同意日志：表结构、写入接口、查询接口。
- [ ] 个人数据导出：用户资料、地址、订单、评价、偏好、同意记录。
- [ ] 账号删除或注销：软删除、PII 匿名化、订单留存说明。
- [ ] Footer/设置页/注册页的法律文档入口。

完成条件：

- 用户首次注册和 Cookie 同意均有服务端留痕。
- 用户可查看同意历史，可申请导出和删除账号。
- 合规文案不承诺当前系统没有实现的数据权利流程。

## Phase L3: 运行时基础设施

优先级：P0
建议周期：1 周
目标：让系统具备可部署、可配置、可恢复的运行基础。

代办清单：

- [ ] 数据库连接池生产配置：连接数、超时、空闲回收、慢查询观察项。
- [ ] 生产日志配置：级别、文件滚动、traceId、敏感字段脱敏。
- [ ] 健康检查升级：应用健康、数据库健康、外部依赖健康。
- [ ] 备份方案：MySQL 全量/增量备份、保留周期、备份加密、备份告警。
- [ ] 备份恢复演练：恢复到临时库并记录结果。
- [ ] 对象存储准备：商品图、聊天图片、数字资源迁移方案和回滚策略。
- [ ] 数据生命周期：搜索日志、旧订单、删除账号、媒体文件清理策略。

完成条件：

- 新环境可按文档从零启动。
- 备份恢复至少演练一次并留下任务记录。
- 日志能定位请求但不泄露敏感信息。

## Phase L4: 性能、缓存与容量

优先级：P1
建议周期：1-2 周
目标：为上线流量建立性能基线、优化点和降级方案。

代办清单：

- [ ] 性能基线：商品列表、搜索、详情、订单、后台列表、推荐、聊天、通知。
- [ ] 数据库索引审查：高频查询 explain、慢查询候选、分页稳定性。
- [ ] Redis 缓存候选：热搜、推荐、商品详情、分类、配置项。
- [ ] 缓存失效策略：商品变更、活动变更、搜索治理规则变更。
- [ ] 异步日志：搜索日志、行为埋点、审计外的非关键统计写入。
- [ ] HTTP 缓存头：静态资源、公开只读接口、禁止缓存的用户接口。
- [ ] 限流策略：登录、注册、搜索、支付、客服发送、后台高风险操作。
- [ ] 压测目标：500 并发用户、p95 < 500ms、错误率 < 1%。

完成条件：

- 有基线和优化后指标对照。
- 缓存、限流、异步写入都有降级策略。
- 压测脚本和结果可复现。

## Phase L5: 容器化与部署流水线

优先级：P0
建议周期：1 周
目标：形成可重复部署、可回滚、可验证的发布路径。

代办清单：

- [ ] Backend Dockerfile。
- [ ] Frontend Dockerfile + Nginx 配置。
- [ ] `docker-compose.yml` 或演示环境 compose 文件。
- [ ] `application-prod.yml` 示例和环境变量说明。
- [ ] 前端生产环境变量：API base、标题、开关配置。
- [ ] CI/CD 部署流程：构建、测试、镜像、部署、健康检查。
- [ ] 数据库初始化策略：schema、seed、生产初始化脚本边界。
- [ ] 回滚方案：镜像回滚、数据库回滚、配置回滚、前端静态资源回滚。

完成条件：

- 新机器可以按 runbook 部署。
- 发布失败时能回滚到上一稳定版本。
- CI/CD 不依赖本机私有路径或手动状态。

## Phase L6: 监控、告警与运维文档

优先级：P1
建议周期：1 周
目标：上线后能发现问题、定位问题、交接问题。

代办清单：

- [ ] 应用指标：请求量、错误率、延迟、JVM、数据库连接池。
- [ ] 业务指标：注册、登录、下单、支付、退款、客服队列、举报、调解。
- [ ] 日志检索：按 traceId、用户、订单号、支付单号定位。
- [ ] 告警规则：接口错误率、支付失败率、数据库连接耗尽、磁盘空间、备份失败。
- [ ] 运维 runbook：部署、回滚、备份恢复、故障排查、密钥轮换。
- [ ] 应急预案：支付异常、订单状态异常、数据库不可用、前端白屏、权限误配。

完成条件：

- 每个 P0 告警都有负责人、阈值和处理步骤。
- 常见故障可以按 runbook 复现排查路径。
- 监控面板不依赖开发者本地环境。

## Phase L7: 上线验收与发布演练

优先级：P0
建议周期：3-5 天
目标：冻结上线候选版本，完成发布前最后验证。

代办清单：

- [ ] 全链路 smoke：注册、登录、认证、浏览、收藏、购物车、下单、支付、履约、评价、客服、举报、调解、后台处理。
- [ ] 权限 smoke：普通用户、店主、客服、审核员、运营、订单管理员、超级管理员。
- [ ] 安全验收：依赖扫描、配置扫描、敏感信息检查、CORS、JWT、权限回归。
- [ ] 合规验收：隐私政策、用户协议、Cookie 同意、用户权利入口。
- [ ] 性能验收：压测结果达到目标或记录上线阻断。
- [ ] 备份恢复验收：最近一次备份可恢复。
- [ ] 发布演练：部署、健康检查、回滚、再部署。
- [ ] 上线 checklist 签收：阻断项关闭，可延期项有负责人和日期。

完成条件：

- 发布候选版本有完整测试记录。
- 所有 P0 blocker 关闭或明确不允许上线。
- 发布、回滚、恢复和故障排查均有可执行文档。

## 推荐实施顺序

1. L0 范围冻结与任务拆分。
2. L1 安全加固和 L2 隐私合规并行启动。
3. L3 运行时基础设施与 L5 容器化部署并行推进。
4. L4 性能容量在可部署环境稳定后开始。
5. L6 监控告警在基础设施和部署路径稳定后接入。
6. L7 上线验收必须等待功能冻结、部署路径和监控备份都完成。

## 与功能路线的协作点

- 功能路线的 F0 契约断点补齐完成后，L7 smoke 才能覆盖完整用户路径。
- 支付升级 F2 完成后，L1 安全验收和 L7 支付/退款 smoke 需要同步更新。
- 数据可视化 F3 可复用 L4 的性能基线和 L6 的监控指标，但业务图表不替代运维监控。
- 个性化 F4 与 L2 隐私设置要保持边界：偏好设置是产品功能，同意日志和用户权利是合规能力。
