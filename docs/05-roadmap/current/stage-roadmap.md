# 当前阶段路线图

本文件只记录“当前阶段正在做什么、下一步做什么、当前风险是什么”。

它不是项目历史总结，也不负责记录已经完成的早期阶段细节。

## 当前阶段

当前阶段定义为：

- 核心交易链路已完成
- 热搜、评价、推荐等主要扩展能力已基本落地
- 项目进入“后 MVP 扩展与工程治理并行”阶段

这一阶段的重点不再是补齐最初骨架，而是：

1. 收紧文档与规范体系，避免 roadmap 与代码现实再次脱节
2. 完成尚未收尾的体验与工程治理项
3. 为聊天 MVP 和平台调解建立可执行任务规格
4. 处理影响后续开发效率的横切工程问题

## 当前优先级

### P1

- 修正文档现实与代码现实的偏差，尤其是当前路线图、数据库运行基线、API 规范状态
- 建立聊天 MVP 的正式任务规格
- 明确聊天与平台调解的边界、依赖与最小实现范围

### P2

- 完成前端包体积第二轮治理
- 收口评价入口、偏好主题能力、后台治理动作一致性等当前 active 任务

### P3

- 在聊天 MVP 边界明确后推进平台调解边界定义
- 在 UI/UX 方案确定后，按新增或调整的接口增量维护 API 规范与 HTTP 验证资产

## 下一阶段入口条件

只有在以下条件满足后，才适合把“聊天/调解”作为主线推进：

- 当前文档体系基本稳定
- 聊天 MVP 与平台调解边界已通过正式任务规格明确
- 评价、推荐、热搜治理相关行为不再频繁返工
- 当前测试与联调资产能支撑继续扩展

## 当前风险

- 文档现实与代码现实容易再次脱节
- 正式 API 规范流程已经形成，但后续 UI/UX 调整仍可能带来接口与文档同步成本
- 聊天与调解一旦同时推进，容易把任务边界再次搅乱
- 前端包体积与管理页增长会持续抬高维护成本

## 使用规则

- 只记录当前阶段与下一阶段相关内容
- 已经失去当前指导价值的阶段说明，移入 `../archived/`
- 具体执行步骤统一下沉到 `../../08-tasks/`
