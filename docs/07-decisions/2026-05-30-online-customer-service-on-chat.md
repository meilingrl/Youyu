# ADR-003: Online Customer Service on the Existing Chat System

**Date:** 2026-05-30
**Status:** Accepted

## Context

The platform needed an in-app "在线客服" (online customer service) capability: users contacting a platform agent, a rule-based assistant answering common campus-marketplace questions, escalation to a human, and an admin console for agents — replacing the old `/admin/support` ticket queue page as the primary CS surface.

Constraints:
- Reuse existing infrastructure; no websockets, no external LLM, no new UI libraries (8s polling like the rest of chat).
- Keep the durable support-ticket system intact (history/fallback).
- Idempotent, backward-compatible schema changes; the repository uses `CREATE TABLE IF NOT EXISTS` with no migration tool, and `CLAUDE.md` forbids `ALTER TABLE` in `schema.sql`. Standard MySQL does not support `ADD COLUMN IF NOT EXISTS` either.

Options considered: (a) a separate realtime CS channel/table set; (b) reuse the chat tables (`chat_conversations`/`chat_messages`) with a new `support` conversation type and a few state columns.

## Decision

Build online CS on top of the existing chat system. Support conversations use `type = 'support'`, always pairing the requester (`user_a`) with a seeded `platform_cs` account (`user_b`); admins reply on its behalf. Two state columns were added to `chat_conversations`: `support_status` (`ai` → `pending` → `human` → `closed`) and `assigned_admin_id`. A rule-based FAQ component (`SupportFaqKnowledgeBase`) auto-replies while `support_status = 'ai'`; escalation flips to `pending` and stops the bot; admins claim (`human`) and close (`closed`) from a new console under `/api/admin/support/chat/*`.

For schema additivity we add the new columns inline to the existing `CREATE TABLE IF NOT EXISTS chat_conversations` (consistent with how prior chat columns were introduced) and guard reads/writes with the existing `columnExists(...)` availability pattern, rather than using `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` (forbidden in `schema.sql` and not portable to MySQL).

## Consequences

**Positive:** Reuses chat polling, message storage, unread counts, and bubble rendering. The platform-CS-as-`user_b` model makes unread tracking (`unread_count_b` = pending agent work) and multi-agent claim/assignment fall out naturally. The ticket system is untouched.

**Negative:** Support state is denormalized onto the shared `chat_conversations` table, so support-only columns are nullable for non-support rows. Pre-existing databases created before this change will not auto-gain the new columns (the `columnExists` guard degrades support features gracefully) until the schema is recreated/seeded.

**Neutral:** A dedicated `platform_cs` user row is required (seeded in `data.sql`); resolving it by username is cached in the mapper. The message-center 客服 tab now keys off `type = 'support'` instead of the previous `type === 'direct'` heuristic.
